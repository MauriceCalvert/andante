# Voice Generation

## Status

v1.0.0 | Normative | Supersedes bass_writer.py, soprano_writer.py, bass_viterbi.py

---

## One Algorithm

Every voice is generated by the same function. No voice-specific
generators exist. The differences between a bass line and a soprano
melody are not algorithmic — they are parametric.

```
generate_voice(
    structural_knots,    # this voice's schema degrees, pinned (may be empty)
    rhythm_grid,         # this voice's onset/duration pairs
    existing_voices,     # all voices composed so far (pitch per beat)
    range,               # instrument actuator range
    cost_weights,        # style, affect, phrase position
    key_info,            # current key context
    imitation_params,    # optional: voice to follow + transform
) → notes
```

The function is a Viterbi solver. At each beat it considers every legal
pitch, scores it against the cost function, and finds the globally
optimal path through all beats simultaneously.

---

## Composition Order

Voices are composed sequentially. Each voice is generated against
all previously composed voices.

Order is determined by **dependency first, then genre preference**.
The dependency graph defined in `voices.md` is absolute: an imitative
voice cannot be composed before the voice it follows. Within the
freedom left by dependencies, genre determines the preferred order:

| Genre | Preferred order | Rationale |
|-------|----------------|-----------|
| Galant | bass → soprano → inner | Bass establishes harmonic foundation |
| Fugue | entering voice → countersubject → free | Subject defines the material |
| Chorale | soprano → bass → tenor → alto | Melody is given |

If a dependency contradicts the genre preference — e.g. a fugue
where the bass enters first as subject — the dependency wins.

The first voice composed has no existing voices to evaluate against.
Its cost function is purely melodic and harmonic (chord tone preference,
stepwise motion, range centering). Each subsequent voice adds pairwise
terms against all predecessors.

## Structural Skeleton

Before any surface is composed, the schema layer places structural
tones for all voices: degree sequences, cadence targets, phrase
boundaries. This is the harmonic architecture — quick, cheap, no
Viterbi. It produces the knots that pin the Viterbi path.

The skeleton exists so that voices composed at different times share
a coherent harmonic plan. Voice B's structural tones were chosen
knowing voice A's structural tones. The surface generation honours
that plan while filling the space between knots with idiomatic motion.

**Knots may be empty.** A `harmony_fill` voice (inner voice derived
from chord voicing) has no schema degrees of its own. It receives
an empty knot list. The solver still works — range, cost weights,
and existing voices provide all the guidance needed. The solver
finds the optimal path through the full pitch space at every beat
without long-range anchor constraints. If long-range anchoring
proves necessary for inner voice quality, synthetic knots can be
derived (e.g. chord thirds at cadence points) without changing
the algorithm.

## Cost Function

The cost function is a weighted sum of independent terms. Each term
is a float. The weights are the style.

### Melodic terms (this voice alone)

| Term | Penalises |
|------|-----------|
| `step_preference` | Intervals larger than a step |
| `leap_penalty` | Leaps, graduated by size |
| `consecutive_leap` | Two leaps in the same direction |
| `range_centering` | Distance from voice's median pitch |
| `repeated_pitch` | Same pitch twice in succession |
| `contour` | Deviation from target melodic shape |

### Pairwise terms (this voice against each existing voice)

Evaluated once per existing voice, then summed.

| Term | Penalises |
|------|-----------|
| `dissonance` | Non-consonant intervals, graduated by beat strength |
| `parallel_perfect` | Parallel fifths and octaves by similar motion |
| `direct_perfect` | Hidden fifths/octaves (similar motion into perfect interval) |
| `voice_crossing` | Voices occupying each other's register |
| `spacing` | Voices too close or too far apart |
| `motion_preference` | Reward for contrary motion, mild penalty for parallel |

### Imitation terms (only when `imitation_params` is set)

| Term | Penalises |
|------|-----------|
| `imitation_fidelity` | Deviation from the followed voice's pitch at (t − delay), after transformation (transposition, inversion) |
| `imitation_rhythm` | Deviation from the followed voice's rhythm pattern at (t − delay) |

Imitation fidelity is a soft cost, not a hard constraint. A strict
canon sets it very high; a free imitative voice sets it low enough
that the solver can depart when counterpoint demands it. A real
fugue answer alters intervals to stay in key — the cost function
naturally produces this behaviour when `imitation_fidelity` competes
with `dissonance` and key membership.

### Vertical terms (this voice + all existing voices together)

These appear at N≥3.

| Term | Penalises |
|------|-----------|
| `chord_completeness` | Missing chord tones when enough voices exist to complete the triad |
| `doubling` | Doubling a pitch class already present in two other voices |

### Application

Every raw cost is multiplied by its corresponding weight before
summation. The solver computes:

```
total = sum(raw_cost[term] * weight[term] for term in all_terms)
```

A weight of zero disables a term entirely. A high weight makes it
near-inviolable. This is the mechanism by which style, affect, and
phrase position control the output: they set the multipliers, not
the logic.

### Constraint

The only hard constraint is **range** — the actuator's physical
limits. Everything else is a cost. Voice crossing, strong-beat
dissonance, parallel fifths: all soft, all tuneable.

## Style as Weights

A style is a named set of cost weights.

```yaml
B19_bach_invention:
  step_preference: 2.0
  leap_penalty: 4.0
  parallel_perfect: 50.0
  voice_crossing: 15.0
  dissonance_strong: 30.0
  dissonance_weak: 1.0
  contrary_motion_reward: 3.0
  chord_completeness: 8.0
  ...

W41_telemann_fantasia:
  step_preference: 1.5
  leap_penalty: 2.5
  parallel_perfect: 50.0
  voice_crossing: 5.0
  dissonance_strong: 25.0
  dissonance_weak: 0.5
  contrary_motion_reward: 2.0
  chord_completeness: 6.0
  ...
```

Differences between composers are not different algorithms. They are
different numbers in the same vector.

---

## Weight Envelopes

Cost weights are not static across a piece. They vary with:

- **Phrase position**: opening (stable, high consonance weight) →
  middle (exploratory, lower penalties) → cadential approach
  (directed, high stepwise preference, chromatic tolerance)

- **Affect/character**: freudigkeit raises leap tolerance and
  widens spacing preference; traurigkeit tightens stepwise
  preference and raises dissonance tolerance for suspensions

- **Formal position**: exposition (strict, high parallel penalty) →
  development (freer, lower penalties) → recapitulation (strict again)

- **Genre**: gavotte weights beat 3; sarabande weights beat 2;
  gigue raises leap tolerance throughout

The rhetorical layer — already present in the system as `character`,
`rhythm_profile`, `affect` — maps to a cost weight envelope that
modulates the base style weights across time. The solver receives
different weights at different beats. The algorithm is unchanged.

```
Rhetorical layer (affect, arc, genre, phrase position)
        ↓
Style map: (affect, position, genre) → cost weight snapshot
        ↓
Viterbi solver (one algorithm, variable weights)
        ↓
Voice surface
```

---

## Learnability

The cost weights are a finite vector of floats. Given a corpus of
composed music with extractable structural skeletons, the optimal
weights for a given composer or style are learnable by minimisation:

> Find weights W that minimise the divergence between the Viterbi
> path under W and the composer's actual pitch choices.

This reconnects the ML pipeline with the rule-based architecture.
The rules are the Viterbi graph structure (what transitions exist).
The learning is the cost weights (which transitions are preferred).
They are not separate systems — they are two aspects of one mechanism.
